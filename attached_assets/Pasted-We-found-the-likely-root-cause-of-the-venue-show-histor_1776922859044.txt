We found the likely root cause of the venue show history bug.

BUG:
When a user does a calc and it auto-saves, the run is saved to history, but it is not being linked to the venue profile.
Venue detail/history reads from runsTable where runsTable.venueId = venue.id.
Right now the frontend creates/updates the venue, but does NOT reliably pass venueId into the saved run payload.

Please fix this end-to-end.

GOAL:
Any calc saved from the single show calculator must appear in:
1. Saved calculations / run history
2. The linked venue profile show history

ROOT CAUSE TO FIX:
- Venue detail uses runsTable.venueId to load venue history
- runs route simply saves what it is given
- run-form currently calls createOrUpdateVenue, but does not use the returned venue.id in the final createRun/updateRun payload
- VenueSearch already supports venueId for selected past venues
- runSelectedVenueId exists in run-form state but is not being properly persisted into the run payload

IMPLEMENT THESE CHANGES:

1. FRONTEND: persist venueId in run-form
- In artifacts/gig-trail/src/pages/run-form.tsx:
  - When a venue is selected from VenueSearch, store the selected venueId in state
  - If user selects an existing venue from the venue database, use that venueId directly
  - If user types/manual-enters a venue and createOrUpdateVenue is called, capture the returned venue object and set venueId = returnedVenue.id
  - When venue is cleared, clear runSelectedVenueId
  - Include venueId in BOTH create and update run payloads
  - Also include vehicleId if there is already a selected vehicle state available and it should persist

2. FRONTEND: make venue selection robust
- Ensure the VenueSearch onSelect handler in run-form preserves:
  - venueName
  - destination
  - city/suburb/state/country
  - lat/lng
  - venueId
- If a manual venue is entered and then saved via createOrUpdateVenue, make sure the resulting venueId replaces null before the run save happens

3. BACKEND SAFETY NET

  - Add a fallback so that if a run is being created/updated with venueName but no venueId, the API attempts to resolve venueId by matching the user’s venue record using normalized venue name
  - Reuse the same normalization logic used in venues.ts if possible
  - If a match is found, inject venueId before save
  - This is a safety net only; frontend should still send venueId properly
